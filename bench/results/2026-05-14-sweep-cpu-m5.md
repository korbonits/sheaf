# 2026-05-14 — Apple M5 CPU sweep

Sustained-RPS sweep across the same three frameworks as the
[50 RPS single-point run](./2026-05-14-cpu-m5.md). Single replica,
single worker process, same model + batch params, same machine. Same
methodology as the [2026-05-09 M1 sweep](./2026-05-09-sweep-cpu-m1.md);
new hardware.

## Setup

| Field | Value |
|---|---|
| Hardware | Apple M5 (10 cores), macOS 26.4 |
| Python | 3.12.13 |
| Model | `amazon/chronos-bolt-tiny` (~80 MB) |
| Workload | 64-step history → 12-step horizon, `output_mode=mean` |
| RPS levels | 10, 50, 100 (sustained, open-loop) |
| Warmup | 5 s per run |
| Measurement window | 20 s per run |
| Batch params | `max_batch_size=8`, `batch_wait=10ms` (all three) |
| Replicas / workers | 1 / 1 (all three) |

## What this shows

### Four findings

**1. M5 has substantially more headroom than M1.** Every framework's
p50 dropped at every RPS level. Raw Ray Serve and BentoML now sustain
100 RPS *without saturation* — on M1 both saturated by 100 RPS. M5's
faster single-core perf clears the chronos-bolt-tiny inference cost
before the batching window closes, so requests don't queue.

**2. sheaf-serve saturates at 100 RPS while raw Ray Serve does not.**
This is the new finding M5 surfaces and M1 couldn't show. On M1, both
Ray-Serve-based servers saturated together at 100 RPS, so the bench
couldn't distinguish "Ray Serve hits its limit" from "sheaf adds enough
overhead to push past the limit." On M5 with more capacity, the
distinction is visible: at 100 RPS the raw Ray Serve baseline holds
p50=18 ms / p99=27 ms; sheaf goes into queue-and-drain saturation
(p50=216 ms, p95/p99 in the seconds).

   The sheaf hot path is "Ray Serve + Pydantic request/response
   validation + `_SheafDeployment` orchestration." The 50 RPS row shows
   that overhead is ~1.3 ms p50 in the sustainable regime. At the edge
   of single-replica capacity, that per-request cost is enough to push
   the framework over the line. This is a **honest regression to
   investigate**: at sub-saturation the typed-contract layer is
   effectively free; at saturation it compounds. Re-running the 100
   RPS sheaf point twice produced p50 = {19.2, 216.3} ms with p95 in
   the seconds both times — the median is volatile, the tail is
   consistently bad.

   What to look at next: per-request CPU profile of `_SheafDeployment.predict`
   vs the raw deployment's predict, and whether the `bucket_requests` /
   `@serve.batch` interaction adds enough scheduling latency to matter
   at the edge.

**3. sheaf-serve and raw Ray Serve are at parity at sustainable RPS.**
At 10 RPS: 34.8 vs 20.3 ms p50 (the gap looks big but absolute scale is
small; both pay the 10 ms batch-wait at low fill rates). At 50 RPS:
20.5 vs 19.2 ms p50 — the 1.3 ms gap is the actual typed-contract cost
in the steady-state regime, and it's small.

**4. BentoML's p50 still drops as RPS climbs.** 8.8 → 5.6 → 4.2 ms.
Same batching-amortisation story as M1: at low RPS most batches don't
fill, every request pays the 10 ms `max_latency_ms`; at high RPS
batches actually fill and per-request overhead drops. Notably, M5
BentoML has **zero errors at 100 RPS** (vs 1.2% drops on M1) — its
single-process queue has clear headroom on this chip.

### What this means for picking a substrate

Same trade-off picture as M1; M5 just moves the inflection points
higher:

| If you need… | Pick |
|---|---|
| Lowest single-replica latency at any RPS this bench tests | BentoML |
| Headroom past M1's 100 RPS saturation point | Raw Ray Serve > sheaf at the edge |
| Horizontal scale-out across multiple nodes | Ray Serve / sheaf |
| Hot-swap deploys, multi-replica orchestration | Ray Serve / sheaf |
| Same code path on Modal serverless and Ray cluster | sheaf |
| Bring-your-own typed contracts for 20+ model types | sheaf |

For a CPU-bound forecast model on a single laptop, BentoML still wins.
For a multi-replica GPU diffusion deployment with hot-swap and Modal
fallback paths, sheaf is still the right tool — and the typed-contract
overhead matters less the more expensive the model call is.

| Framework | RPS | M1 p50 | M5 p50 | Note |
|---|---:|---:|---:|---|
| sheaf | 10 | 31 | 34.8 | comparable |
| sheaf | 50 | 32 | 20.5 | 1.6× faster |
| sheaf | 100 | 44,655 | 216.3 † | both saturated, M5 less so |
| Raw Ray Serve | 10 | 29 | 20.3 | 1.4× faster |
| Raw Ray Serve | 50 | 33 | 19.2 | 1.7× faster |
| Raw Ray Serve | 100 | 45,186 | 18.0 | M5 does not saturate |
| BentoML | 10 | 15 | 8.8 | 1.7× faster |
| BentoML | 50 | 7 | 5.6 | 1.3× faster |
| BentoML | 100 | 7 (drops 1.2%) | 4.2 (no drops) | M5 has more headroom |

The big delta is BentoML and raw Ray Serve gaining substantial
headroom at 100 RPS on M5 — neither saturates anymore. The fact that
sheaf still struggles there, where its substrate does not, is the
honest finding.

## Honest limitations of this run

- **Single sample per (framework, RPS) point.** p50 at 100 RPS for
  sheaf in particular is volatile across re-runs (see finding 2);
  multiple seeds + medians would be more robust. Single samples are
  enough to spot the divergence, not enough to size it precisely.
- **Single machine, single replica.** Ray Serve's scale-out value
  cannot be shown here.
- **No GPU model.** chronos-bolt-tiny is CPU-friendly; the picture
  changes for GPU-bound workloads where the model call dominates and
  framework overhead is rounding error.

```bash
# Repo root
uv sync --extra dev --extra time-series
uv pip install bentoml

# Per framework: start the server, sweep at 10/50/100 RPS, kill server.
# All servers use the bench fixtures in bench/servers/; loadgen is bench/loadgen.py.
# Per-RPS JSON results land in bench/results/2026-05-14-sweep-cpu-m5/.
```
