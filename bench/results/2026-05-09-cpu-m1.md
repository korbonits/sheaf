# 2026-05-09 — Apple M1 CPU run

Single-machine, CPU-only comparison of sheaf-serve, raw Ray Serve, and
BentoML serving the same model with the same batching parameters. The
bench is the methodology described in [`../README.md`](../README.md);
this file is the data.

## Setup

| Field | Value |
|---|---|
| Hardware | Apple M1 (8 cores), macOS 26.3.2 |
| Python | 3.12.13 |
| Model | `amazon/chronos-bolt-tiny` (~80 MB) |
| Workload | 64-step history → 12-step horizon, `output_mode=mean` |
| Target RPS | 20 (sustained, open-loop) |
| Warmup | 5 s |
| Measurement window | 20 s (~400 samples) |
| Batch params | `max_batch_size=8`, `batch_wait=10ms` (all three) |
| Replicas / workers | 1 / 1 (all three) |
| sheaf-serve | v0.10.0 |
| Ray | 2.50.x |
| BentoML | 1.4.39 |

## Latency (steady-state, post-warmup)

| Framework | p50 (ms) | p95 (ms) | p99 (ms) | Mean (ms) | Max (ms) | Errors |
|---|---:|---:|---:|---:|---:|---:|
| **sheaf-serve** | 25.6 | 34.1 | 43.6 | 26.5 | 54.3 | 0 |
| Raw Ray Serve | 25.9 | 38.7 | 143.5 | 30.0 | 203.2 | 0 |
| BentoML | 13.2 | 16.4 | 29.7 | 13.5 | 45.3 | 0 |

All three sustained 20.05 RPS over 20 seconds (target was 20.00; the
0.25% overshoot is the open-loop tail from the warmup boundary).

## What this shows

**sheaf vs raw Ray Serve — parity, with a tighter tail.** The whole
sheaf path is "Ray Serve + a typed-contract layer", so we expect parity
on p50; we got it (25.6 vs 25.9 ms). The p99 difference (43.6 vs 143.5)
is *favorable* to sheaf in this run — but with only 400 samples a
single GC pause moves the 4th-worst sample by ~100 ms. **Don't read
this run as "sheaf is faster than Ray Serve."** Read it as "the
typed-contract layer is free in expectation, and the tail behaviour is
within run-to-run noise of raw Ray Serve."

**sheaf vs BentoML — BentoML is faster at this RPS, and that's
real.** BentoML's p50/p95/p99 are roughly half of sheaf's. The cause is
the Ray Serve substrate: every request hops `client → HTTP proxy actor
→ replica actor → batched call → reply path`. At 20 RPS most batches
contain a single request, so the `@serve.batch` 10 ms wait is paid in
full per request without any of the amortisation that batching exists
to provide. BentoML's in-process adaptive batching has lower per-hop
overhead at this scale.

This is a real cost of choosing Ray Serve as the substrate. Sheaf
inherits it. The trade is: Ray Serve gives you horizontal scale-out,
hot-swap, and multi-replica orchestration for free; BentoML's
single-process model is faster at low concurrency but does less when
you scale out across nodes. **Pick the substrate that fits the
deployment, not the one that wins a single-replica latency
microbench.**

This benchmark needs a higher-RPS run to show the regime where
batching amortisation flips the picture. Adding it is on the to-do.

## Lines of code per server

A user deploying the same model with the same batching parameters
writes roughly half the code with sheaf-serve. Counts are
non-blank-non-comment SLOC of the example server file checked into
[`../servers/`](../servers/):

| Framework | Total lines | SLOC | vs. sheaf |
|---|---:|---:|---:|
| **sheaf-serve** | 45 | 36 | — |
| Raw Ray Serve | 82 | 58 | +61% |
| BentoML | 72 | 57 | +58% |

The framework-vs-no-framework axis is where sheaf earns its keep:
declare a `ModelSpec`, instantiate `ModelServer`, call `.run()`. No
hand-rolled FastAPI app, no `@serve.deployment`/`@serve.ingress` glue,
no separate batchable function with hand-managed list signatures, no
Pydantic request/response classes per backend.

## Honest limitations of this run

- **Single RPS point.** A real comparison needs a sweep — at minimum
  10/50/100/200 RPS — to show the regime where batching amortisation
  flips the latency story. This run is a sanity check, not a sweep.
- **Single sample of size 400.** p99 with 400 samples is the 4th-worst
  number; a single GC pause moves it. Re-run for stability before
  drawing conclusions about the tail.
- **Single machine, single replica.** Ray Serve's value proposition is
  horizontal scale-out across nodes; an M1 laptop bench can't show
  that, and BentoML's single-process advantage shrinks the moment you
  need a second replica.
- **No GPU model.** Chronos-Bolt-tiny is CPU-friendly; the picture
  changes for GPU-bound workloads where the model call dominates and
  framework overhead is rounding error.

## Reproducing this exact run

```bash
# Repo root
uv sync --extra time-series
uv pip install bentoml

# Sheaf
.venv/bin/python bench/servers/sheaf_server.py &
sleep 90  # wait for chronos download + Ray Serve start
.venv/bin/python bench/loadgen.py --url http://127.0.0.1:8000/forecaster/predict \
    --target-rps 20 --duration 20 --warmup 5 --label sheaf
kill %1; ray stop --force

# Raw Ray Serve
.venv/bin/python bench/servers/ray_serve_server.py &
sleep 60
.venv/bin/python bench/loadgen.py --url http://127.0.0.1:8000/forecaster/predict \
    --target-rps 20 --duration 20 --warmup 5 --label ray-serve
kill %1; ray stop --force

# BentoML  (from bench/servers/ — the bentoml CLI loads bentoml_server:svc)
cd bench/servers
.venv/bin/bentoml serve bentoml_server:svc --host 127.0.0.1 --port 8000 &
sleep 30
cd ../..
.venv/bin/python bench/loadgen.py --url http://127.0.0.1:8000/predict \
    --target-rps 20 --duration 20 --warmup 5 --label bentoml --wire bentoml
kill %1
```
